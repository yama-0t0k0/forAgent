# パスキー共通化のための「ドメイン統一」設計（本プロジェクト実態版）

このドキュメントは、当プロジェクト（engineer-registration-app-yama）の実態（Firebase Hosting が複数site運用）に合わせて、**リリース後の運用まで破綻しにくい** パスキー（WebAuthn）共通化方針をまとめたものです。

結論（先に）:
- **Hosting site は分けたまま**（Admin/LP/Dev などのデプロイ単位を維持）
- **同一の親ドメイン配下にサブドメインを割り当てる**（例: `example.com`, `admin.example.com`）
- **Passkey の RP ID は “親ドメイン” に固定**して共通化する（サブドメインでも共通化可能）
- “API Gateway 的に統一” したい範囲は、まず **APIパス（例: `/api/...`）だけ** を Hosting rewrites で集約する

つまり、「フロントも含めて全部をGatewayに集約」はやらないが、「APIパスのrewrites」は状況により使う、ということです。

この構成は、運用上の事故ポイント（SPAのベースパス対応、静的アセットの参照、OAuthリダイレクト等）を増やさず、将来の Flutter Mobile に必要な `.well-known` 配信も一箇所に寄せやすいです。

------------------------------
## 0. 【現状】本プロジェクトの Hosting 実体（コードベースより）
本プロジェクトは `firebase.json` の `hosting` が配列で、`target` ごとに別siteへデプロイします。

### Hosting targets → sites（.firebaserc）
- `admin-app` → `admin-app-site-d11f0`
- `lp-domain` → `engineer-registration-lp`
- `lp-domain-dev` → `engineer-registration-lp-dev`

### 現在の配信ドメイン（Firebase標準）
- Admin: `https://admin-app-site-d11f0.web.app` / `https://admin-app-site-d11f0.firebaseapp.com`
- LP: `https://engineer-registration-lp.web.app`
- LP Dev: `https://engineer-registration-lp-dev.web.app`

------------------------------
## 1. 【ドメイン戦略】親ドメインを決め、RP ID を親ドメインへ固定する
パスキー（WebAuthn）の共通化に “パス統合” は必須ではありません。  
重要なのは **RP ID（Relying Party ID）を、全入口（Admin/LP）で共通の “親ドメイン” に固定する**ことです。

推奨するカスタムドメイン設計例:
- LP: `https://example.com`（LP site を割当）
- Admin: `https://admin.example.com`（Admin site を割当）
- Dev（必要なら）: `https://dev.example.com`（Dev site を割当）

Passkey の RP ID:
- `example.com`（親ドメイン）に固定

補足（誤解が出やすい点）:
- 「サブドメインだとパスキー共通化できない」は誤りです。RP ID を親ドメインにすれば共通化できます。

------------------------------
## 2. 【入口/API集約】Hosting rewrites で “APIだけ” を寄せる（現実解）
「フロントも含めて 1site にまとめ、`/admin` `/lp` のようにパスで切る」方式は、SPA のベースパス対応やアセット参照の調整が必要になり、移行コストと運用リスクが高いです。

当プロジェクトでまず効果が大きく、導入リスクが小さいのは **APIだけを `/api/...` に集約**する方法です。

狙い:
- Admin/LP など **どの hosting site からでも** `https://{domain}/api/...` で同じバックエンドを呼べる
- 将来の Flutter（Web/Mobile）で、接続先URLを “1つの基底” に寄せやすい

### 例: Cloud Run へ `/api/matching/**` を転送（複数targetに同一設定）
`firebase.json` は hosting が配列なので、必要な入口（target）それぞれに rewrites を入れます。

```json
{
  "hosting": [
    {
      "target": "admin-app",
      "rewrites": [
        {
          "source": "/api/matching/**",
          "run": { "serviceId": "matching-functions", "region": "asia-northeast1" }
        },
        { "source": "**", "destination": "/index.html" }
      ]
    },
    {
      "target": "lp-domain",
      "rewrites": [
        {
          "source": "/api/matching/**",
          "run": { "serviceId": "matching-functions", "region": "asia-northeast1" }
        }
      ]
    }
  ]
}
```

------------------------------
## 3. 【要注意】“1siteに全部パス統合” をやるなら、ベースパス対応が必須
どうしても「フロントも含めて 1ドメイン1site に統合し、`/admin` `/lp` `/api` で切る」方式を採る場合、以下が必須になります。

- Admin（SPA）が `/admin/` をベースパスとして正しく動く
  - ルーティング（pushState）
  - 静的アセット参照パス
  - リダイレクトURL（OAuth等がある場合）
- バックエンド（Functions/Run）が `/api/...` 配下で動く（rewrite/実装の両面で整合）

現状のプロジェクト運用（複数site前提）では、まずは **サブドメイン分割 + 親ドメインRP ID固定**を推奨します。

------------------------------
## 4. 【パスキー】RP ID / expectedOrigin の管理（ここが最重要）
サーバ側（Cloud Functions）でパスキーの登録・検証を行う際、次の2点が致命的に重要です。

- **RP ID（Relying Party ID）**: 親ドメインに固定（例: `example.com`）
- **expectedOrigin**: 実際の配信オリジン（例: `https://example.com`, `https://admin.example.com`）を許可リストで管理

推奨: Functions の環境変数で管理する
- `PASSKEY_RP_ID=example.com`
- `PASSKEY_RP_IDS='["example.com"]'`
- `EXPECTED_ORIGINS='["https://example.com","https://admin.example.com","ios:bundle-id:com.engineer.registration.lpapp"]'`

移行期間の互換運用:
- 既存の `web.app` / `firebaseapp.com` も一時的に許可し、段階的に allowlist を絞る

推奨: Web（Admin/LP）側は build-time の env で RP ID を固定する
- `EXPO_PUBLIC_PASSKEY_RP_ID=example.com`

------------------------------
## 5. Flutter Mobile 移行のための重要ファイル（配信場所の方針）
将来、iOS/Androidアプリでパスキー（関連ドメイン連携）を成立させるため、以下を配信します。

- Android: `/.well-known/assetlinks.json`
- iOS: `/.well-known/apple-app-site-association`

運用上の推奨:
- これらは **親ドメイン側（例: `https://example.com` = LP site）に集約**して配信する  
  （ドメインが分散すると、更新漏れ・TTL/キャッシュ・設定差分の事故が増えます）

------------------------------
## 6. 次のステップ（運用導入の順序）
1. カスタムドメインの追加（Firebase Hosting の site へ割当）
   - `example.com` → LP site
   - `admin.example.com` → Admin site
   - `dev.example.com`（必要なら）→ Dev site
2. Passkey 設定の本番化（Functions env / Web env）
   - `PASSKEY_RP_ID` を親ドメインに固定
   - `EXPECTED_ORIGINS` に配信オリジンを列挙
   - Webビルドに `EXPO_PUBLIC_PASSKEY_RP_ID` を設定
3. 移行期間の互換運用 → 収束
   - 旧ドメイン（`web.app` 等）を allowlist に残し、問題がなければ除外
4. 監視とロールバック設計
   - Passkey 登録/検証の失敗ログ（origin/rpId）を観測し、ドメイン移行時の事故を即検知できるようにする

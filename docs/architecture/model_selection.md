# 専門エージェント用ローカル LLM 選定記録

本ドキュメントは、自律型エージェント（IronClaw）の思考エンジンとして採用するローカル LLM の選定経緯と、リソース制約に基づく決定事項を記録します。

## モデル選定の変遷

### 1. Gemma 4 (E4B)
- **ステータス**: 推奨（ハイスペック環境用）
- **経緯**: Google の最新モデルであり、ツール実行（Function Calling）能力に優れている。
- **断念の理由（16GB環境）**: モデルサイズが 9.6GB と大きく、16GB RAM の Mac では他の開発ツール（Expo, VS Code 等）と並走させた際にメモリ枯渇（スワップ発生）を引き起こし、HTTPリクエストのタイムアウトが多発したため。

### 2. Gemma 2 (2B)
- **ステータス**: 不採用
- **経緯**: メモリ負荷を抑えるために 1.6GB の最軽量モデルとして試行。
- **不採用の理由**: 現在の Ollama 環境 (v0.20.7) において、Gemma 2 2B は `completion` 能力のみであり、**「ツール実行（Function Calling）」の機能フラグが有効ではなかった**ため、IronClaw による自律操作が不可能であった。

### 3. Qwen 2.5 (3B)
- **ステータス**: **現行標準（16GB環境用）**
- **採用の理由**:
  - **軽量性**: 1.9GB とコンパクトで、16GB RAM の環境でも安定して常駐可能。
  - **ツール対応**: `ollama show` にて `Capabilities: tools` を明示的にサポートしており、IronClaw からのファイル操作やコマンド実行の指示を理解できる。
  - **推論性能**: コーディング関連の指示に対する日本語理解と構造化出力の精度が高く、自律開発サイクルを回すのに十分な実力を持つ。

### 4. 専門エージェント・チームトポロジー
チームトポロジーの哲学に準拠した 7 つの専門エージェント
.agent/skills/配下に定義されている。


## 連携の仕組み

IronClaw、Qwen、および 各専門家エージェント の役割分担と連携について、わかりやすく解説いたします。

今回の仕組みは「中央集権的な推論」と「専門家による実作業」を組み合わせたハイブリッド構成になっています。

各コンポーネントの役割
Qwen 2.5 3B (Local LLM / 戦略家)

役割: 「高度な思考と判断」を担当します（Ollama経由で動作）。
具体的な仕事:
GitHub Issue を読み込み、それを**高粒度なタスク分割（DAG：作業グラフ）**へと分解します。
もしタスクが失敗した場合、そのログ（実行結果）を読み取って**「なぜ失敗したのか」「どう修正すべきか」を分析**し、次のアクションを決定します。
IronClaw (実行エンジン / 手足・防波堤)

役割: エージェントが安全にコードを読み書きするための「実行環境（Sandbox）」を提供します。
具体的な仕事:
指示を受けて専門家エージェントを起動します。
エージェントが「許可されたフォルダ以外を壊さないか」「変なコマンドを実行しないか」を監視・隔離する役目も果たします。
専門家エージェント (apps_expert, platform_shared等 / 職人)

役割: 特定の領域に特化した「コーディング作業」を担当します。
具体的な仕事:
.agent/skills/*.md に定義された自分の専門知識（例：UIデザイン、共通基盤、アプリロジック）に基づき、実際にファイルを修正します。
作業はすべて IronClaw の保護下で行われます。
今回の実行（Issue #86）での連携フロー
【計画フェーズ】
オーケストレーターが Qwen を呼び出し、「Issue #86 を確実にこなすための10個の小さなタスクを作ってくれ」と頼みます。
【作業委譲フェーズ】
分割されたタスク（例：ボタンの配置）に対し、オーケストレーターが IronClaw を介して最適なエージェント（例：apps_expert）を召喚します。
コマンド例: ironclaw run -n apps_expert -m "ステップ1: 応募ボタンを配置せよ"
【実行・監視フェーズ】
apps_expert が作業し、その様子（ログ）を Watchdog が 1秒ごとに監視します。
もし無限ループやエラーが発生したら、Watchdog が即座にターミナルを赤く光らせて知らせます。
【分析・修復フェーズ】
万が一失敗したら、再び Qwen が登場し、「ログを見たところ、Firestore の権限でエラーが出ています。次は権限を修正するタスクを追加しましょう」と戦略をアップデートします。
まとめると：

Qwen が「何をすべきか」を考え、
IronClaw が「安全な作業場」を確保し、
専門家エージェント が「実際のコード」を書きます。
この連携により、これまでの「何が起きているかわからないまま止まる」状態から、**「失敗しても原因を自分で分析し、人間に報告して止まる（あるいは修正に挑む）」**という粘り強いシステムに進化しました。

## 今後の指針

1. **メモリ 32GB 以上の環境**: 実装の複雑度が増した場合は、Gemma 4 等のより大規模なモデルへのアップグレードを再検討する。
2. **Apple Silicon への最適化**: モデルの量子化ビット数や `num_ctx`（コンテキスト長）を調整し、16GB 環境下での推論速度と安定性のバランスを継続的に最適化する。
3. **モデル非依存**: オーケストレーター（`pm_orchestrator.js`）はモデル名を動的に変更可能な設計を維持し、より優れた軽量モデルが登場した際に即座に切り替えられるようにする。

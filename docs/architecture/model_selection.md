# 専門エージェント用ローカル LLM 選定記録

本ドキュメントは、自律型エージェント（IronClaw Autonomous Core）の思考エンジンとして採用するローカル LLM の選定経緯と、リソース制約に基づく決定事項を記録します。

## モデル選定の変遷

### 1. Gemma 4 (E4B)
- **ステータス**: 推奨（ハイスペック環境用）
- **経緯**: Google の最新モデルであり、ツール実行（Function Calling）能力に優れている。
- **断念の理由（16GB環境）**: モデルサイズが 9.6GB と大きく、16GB RAM の Mac では他の開発ツール（Expo, VS Code 等）と並走させた際にメモリ枯渇（スワップ発生）を引き起こし、HTTPリクエストのタイムアウトが多発したため。

### 2. Gemma 2 (2B)
- **ステータス**: 不採用
- **経緯**: メモリ負荷を抑えるために 1.6GB の最軽量モデルとして試行。
- **不採用の理由**: 現在の Ollama 環境 (v0.20.7) において、Gemma 2 2B は `completion` 能力のみであり、**「ツール実行（Function Calling）」の機能フラグが有効ではなかった**ため、IronClaw による自律操作が不可能であった。

### 3. Qwen 2.5 (3B)
- **ステータス**: **現行標準（16GB環境用）**
- **採用の理由**:
  - **軽量性**: 1.9GB とコンパクトで、16GB RAM の環境でも安定して常駐可能。
  - **JSON 構造出力の安定性**: コーディング関連の指示に対する日本語理解と構造化出力の精度が高く、自律開発サイクルを回すのに十分な実力を持つ。

## Action Execution (実行ループ) におけるアーキテクチャの決定事項

v3 自律稼働テストの過程で、以下の課題と解決策（方針）が決定しました。これらの決定は、ローカルLLMの制約を克服しつつ、高い信頼性と省エネを実現するためのものです。

### 1. Tool Capability への非依存（Hybrid Adapter 方式）
初期は Ollama API の `chat + tools` エンドポイントに依存する想定でしたが、モデルへの依存度を下げるため不採用としました。
- **課題**: `tools` Capability の挙動はモデルごとに異なり、特定の小規模モデルでは Function Calling が安定しない、または将来APIの仕様変更に対応できなくなるリスクがあった。
- **解決策**: `docs/CodingConventions/tool_call_schema.json` を SSOT（単一の真実の情報源）として定義。LLMには `format: 'json'` で出力を強制し、プロンプトにこのスキーマを埋め込む**ハイブリッド・アダプタ方式**を採用。特定のAPIエンドポイントに依存せず、いかなるLLMでもJSONさえ出力できればツール実行が可能になりました。

### 2. コンテキスト飽和の防止（Two-Stage Prompting）
- **課題**: 以前は「どう計画するか」と「どのツールを実行するか」を毎回ひとつの巨大なプロンプト（Project Context, Milestone, SKILL等を含む）で推論させており、数千トークンの無駄が発生、推論遅延や文脈迷子を招いていた。
- **解決策**: プロンプトを「計画フェーズ（1度だけ重いコンテキストを渡してタスクのDAGを作る）」と「実行フェーズ（対象タスクの指示、JSON Schema、直前のツール結果だけを渡すループ）」の**二段階に分離（Two-Stage Prompting）**し、トークン消費を劇的に削減・安定化しました。

### 3. 不正JSONの物理的遮断と自己修復（Physical Validation & Self-Healing）
- **課題**: プロンプトで「必ずこのJSON Schemaに従え」と指示しても、LLMエスケープ漏れや括弧の閉じ忘れ、未定義フィールドの混入など、不完全なJSONが出力されるリスクが常にある。
- **解決策**: プロンプトでの「お願い（ソフト制約）」に頼るのをやめ、Rust製システム (`ironclaw_core --validate-tool`) による **物理的制約（Hard Constraint）** を導入。`serde_json` の厳格な型システムを用いて実行前にJSONをパースし、構造違反があれば `exit 1` と共に出力されるRustのコンパイラレベルのエラーメッセージをキャッチします。これを直ちにOrchestratorがLLMにフィードバックし、「エラーがあるので修正せよ」と突き返す**自己修復(Self-correction)ループ**を標準装備しました。

## 専門エージェント・チームトポロジー
チームトポロジーの哲学に準拠した 7 つの専門エージェントが `.agent/skills/` 配下に定義されています。


## 連携の仕組み

IronClaw Autonomous Core、Qwen (Local LLM)、および 各専門家エージェント の役割分担と連携について解説いたします。

この仕組みは「IronClaw Core を中心とした自律実行」と「専門家による分業」を組み合わせたハイブリッド構成です。Antigravity（クラウド AI）は、ユーザーの自然言語を Issue に翻訳して投入するところまでが役割であり、以降の実行フェーズには関与しません。

### 各コンポーネントの役割

#### IronClaw Autonomous Core (自律システムの中枢 — Router + Executor + Safety Guard)

**役割**: システム全体の「心臓部」。Issue の検知からタスクの完了報告まで、すべてを自律的に統括します。
「IronClaw Core」は**システム全体の名称**であり、内部的には Orchestrator (Node.js) + Safety Guard (Rust) + Watchdog (Node.js) の 3 コンポーネントで構成されます。

**具体的な仕事**:
- **Orchestrator** (`pm_orchestrator.js`): GitHub Issue をポーリング検知し、処理を開始します。ホスト上で直接実行され、Ollama (localhost:11434) に直結で通信します。
- **Orchestrator → Qwen**: Local LLM (Qwen) を呼び出してタスクの DAG（作業グラフ）を計画させます。
- **Orchestrator → Agents**: DAG の各タスクを、最適な専門家エージェントに割り当てて逐次実行します。
- **Safety Guard** (`ironclaw_core`): 全 LLM 出力を stdin/stdout パイプ経由で Regex 検閲し、機密情報の漏洩を物理的に遮断します。
- **失敗分析**: タスク失敗時に Qwen を再度呼び出し、原因分析と修正タスクの追加を行います。

#### Qwen 2.5 3B (Local LLM / 思考エンジン)

**役割**: IronClaw Core が呼び出す「高度な思考と判断」を担当します（Ollama 経由で動作）。

**具体的な仕事**:
- GitHub Issue を読み込み、**高粒度なタスク分割（DAG：作業グラフ）** へと分解します。
- タスクが失敗した場合、ログ（実行結果）を読み取って **「なぜ失敗したのか」「どう修正すべきか」を分析** し、次のアクションを決定します。

#### 専門家エージェント (apps_expert, platform_shared等 / 職人)

**役割**: 特定の領域に特化した「コーディング作業」を担当します。

**具体的な仕事**:
- `.agent/skills/*.md` に定義された自分の専門知識（例：UIデザイン、共通基盤、アプリロジック）に基づき、実際にファイルを修正します。
- 作業はすべて IronClaw Core の保護下（Podman Rootless Sandbox 内）で行われます。

### 自律実行の連携フロー
1. **【検知フェーズ】** IronClaw Core が `ready-for-agent` ラベルの Issue を検知します。
2. **【計画フェーズ】** Core が Qwen を呼び出し、「この Issue を確実にこなすための N 個の小さなタスクを作ってくれ」と依頼します。
3. **【作業委譲フェーズ】** 分割されたタスク（例：ボタンの配置）に対し、Core が最適なエージェント（例：apps_expert）を Sandbox 内で起動します。
   コマンド例: `ironclaw run -n apps_expert -m "ステップ1: 応募ボタンを配置せよ"`
4. **【実行・監視フェーズ】** 専門家エージェントが作業し、その様子（ログ）を Watchdog が監視します。異常が発生したら即座にアラートを発します。
5. **【分析・修復フェーズ】** 失敗時は Qwen が再度登場し、「ログを見たところ、Firestore の権限でエラーが出ています。次は権限を修正するタスクを追加しましょう」と戦略をアップデートします。

### まとめ

- **IronClaw Core** が「何をすべきか」を計画し、「誰にやらせるか」をルーティングし、「安全に」実行する。
- **Qwen** が Core に呼ばれて「思考と分析」を提供する。
- **専門家エージェント** が Core の指示に従い「実際のコード」を書く。
- **Antigravity** はユーザーの指示を Issue に翻訳するだけ。実行フェーズには介在しない。

この連携により、**クラウドトークンの消費を構造的にゼロに近づけながら**、「失敗しても原因を自分で分析し、人間に報告して止まる（あるいは修正に挑む）」という粘り強い自律システムが実現しています。

## 今後の指針

1. **メモリ 32GB 以上の環境**: 実装の複雑度が増した場合は、Gemma 4 等のより大規模なモデルへのアップグレードを再検討する。
2. **Apple Silicon への最適化**: モデルの量子化ビット数や `num_ctx`（コンテキスト長）を調整し、16GB 環境下での推論速度と安定性のバランスを継続的に最適化する。
3. **モデル非依存**: IronClaw Core（`pm_orchestrator.js`）はモデル名を動的に変更可能な設計を維持し、より優れた軽量モデルが登場した際に即座に切り替えられるようにする。

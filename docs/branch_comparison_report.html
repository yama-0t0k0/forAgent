ブランチ比較レポート

main → yama

プロジェクト構造が単一のExpoプロジェクトから、スケーラブルなモノレポ（Monorepo）構造へと大幅に刷新されました。

main: 単一構成 (Monolith)

ルートディレクトリにすべてのソースコードが集約。小規模開発には適していますが、機能追加に伴う複雑化が懸念される状態。

yama: モノレポ構成 (Monorepo)

Melosを活用し、アプリ・共通ロジック・インフラを分離。複数のアプリケーションを一つのリポジトリで効率的に管理可能。

yamaブランチでの主な進化

1

複数アプリの並行開発

  • • 管理者用 (admin_app)
  • • 企業用 (corporate_user_app)
  • • 個人用 (individual_user_app)
2

共通化の徹底 (Shared)

  • • フロントエンドのUIコンポーネント共通化
  • • Firebase通信・APIロジックの集約
  • • ドメインモデルの型定義共有
3

インフラ・テストの整備

  • • Firebaseエミュレータ設定の統合管理
  • • Firestoreのセキュリティルールと統合テスト
  • • デプロイ自動化スクリプトの充実
4

開発者体験 (DX) の向上

  • • GitHooksによる自動チェック
  • • 共通ドキュメント管理 (docs/)
  • • Melosによる全パッケージ一括操作

結論: プロジェクトの進化

`yama` ブランチへの移行により、プロジェクトは単一のアプリ開発から、複数のサービスを効率的に開発できるスケーラブルなプラットフォームへと進化しました。
コード共有の最大化、一貫したテスト環境、そして明確な責任境界により、今後のチーム開発において高い生産性を維持することが可能です。

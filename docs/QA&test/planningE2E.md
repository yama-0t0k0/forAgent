# E2Eテスト戦略とアーキテクチャ: Golden Path (王道ルート) の確立

## 1. 経緯と課題背景
当初の E2E テストは各機能（例: プロフィール編集フロー）ごとに細かく分割されたシナリオとして存在し、シミュレータ上で単発的に実行されていました。しかし、この手法には以下の課題がありました：
1. **シミュレータ起動の不安定性**: Expo Go の起動、システムダイアログの表示不確実性（UI言語による差分、通知許可のポップアップ等）により、ソースコード自体には問題がないのにテストがタイムアウト・失敗するケースが半数近くを占めていた。
2. **原因特定の困難さ**: テストが失敗した際に、「UIが変わったのか」「通信エラーか」「インポートエラーか」の判別がつかず、開発スピードが著しく低下していた。
3. **断片化**: 小粒度のテストでは、結局「ログインして、操作して、ログアウトする」という業務システムとしての全体的な安心感（Confidence）が得られにくかった。

## 2. 決定した新しいE2Eテスト戦略
これらの課題を解決するため、**「細切れのテストを廃止し、少数の分厚い Golden Path（王道ルート）にリソースを集中させる」** 方針へ転換しました。

### A. Golden Path の採用
各アプリケーション（LP、管理画面、法人、個人）につき、1〜2本の「もっとも重要で長い業務シナリオ」のみを検証します。
- 一連の流れを通してバグが出なければ、システム全体の可用性が保証されるとみなします。

### B. ハイブリッド検証 (Mobile + Web)
シミュレータへの依存（単一障害点）を排除するため、E2Eテストを2層構造にしました。
1. **モバイル検証 (Maestro)**
   - 目的: 実際のiOS/Android環境でのネイティブ挙動確認。
   - 実行: `tests/golden_paths/*.yaml` を使用。OSダイアログ（"Open"/"開く"、"Continue"）も包括的にハンドリングし、堅牢化。
2. **Webローカル検証 (Playwright)**
   - 目的: シミュレータ不要で、ローカルPC上で安全・高速にロジックとUIを通しテストする。
   - 実行: `scripts/test_web_e2e.sh` 経由で実行。実行時間は数秒で完了し、開発中のデバックループを強烈に支えます。

### C. 証拠（Evidence）の絶対的確保
テストが失敗した場合でも、なぜ失敗したかを自律エージェントや開発者がすぐに判別できるよう、以下の証拠保存基盤（Evidence Capture）を確立しました。
- **保存先**: `tests/evidence/`
- **ファイル命名規則**: `[アプリ名]_[YYYYMMDD_HHmmss]_[種別].[拡張子]` （例: `admin_app_20260422_105434_screen_final.png`, `lp_app_web_20260422_113431_playwright.log`）
  - **【徹底事項】**: スクリーンショット画像（.png）だけでなく、**実行ログファイル（.log）やデータ差分ファイル（.json）等の全てのエビデンス**において、人間が直感的に探し出せるよう必ず「何時何分」が判別できるローカルタイムの `YYYYMMDD_HHmmss` フォーマットを含めなければなりません。単なる `Date.toISOString()`(UTC等) による代用や時刻の省略は固く禁じます。
- **収集する証拠**:
  - テスト実行直前・直後の対象ユーザーの **Firestore JSONダンプ（差分用）**
  - エラー発生時および最終画面の **スクリーンショット（自動撮影）**
  - Maestro / Playwright の **詳細実行ログ**

## 3. 実行プロセス (One-Button Execution)
これらの E2E プロセスは、すべて `npm run vet`（`scripts/vet.sh`）に統合され、あるいは必要に応じて `scripts/test_golden_paths.sh` (Maestro用) や `scripts/test_web_e2e.sh` (Playwright用) で個別に実行されます。

これにより、「テストしたい」と思い立ってから数秒〜数分以内で、完全な証明つきのフィードバックが得られる基盤が整いました。

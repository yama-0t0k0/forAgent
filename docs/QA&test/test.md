# 🧪 テスト・品質保証ガイドブック

## 📌 概要
当プロジェクトでは、**「バックエンドにおける鉄壁のセキュリティ」**と**「フロントエンドにおける最高のユーザー体験」**を両立させるため、多層的なテスト戦略を採用しています。

このドキュメントは、開発者が新しい機能を追加したり、セキュリティルールを変更したりする際の、テスト手法と実行手順の決定版ガイドです。

---

## 🛡️ テスト哲学：多層防御 (Defense in Depth)

当プロジェクトのテストは、単なる「バグ探し」ではなく、**「システムへの信頼をコードで証明すること」**を目的としています。

1.  **Backend (Security Rules)**: 
    -   外部からの不正操作、データ漏洩を「ルールレベル」で物理的に遮断します。
    -   フロントエンドのバグが万が一あっても、バックエンドでデータは守られる状態を維持します。
2.  **Frontend (Application Flow)**: 
    -   実際のユーザー操作に基づき、UIの変化と背後のFirestoreデータが正当に更新されているかを検証します。
    -   「操作 -> データ書き込み -> 反映確認」の一連の整合性を保証します。

---

## 🔒 1. バックエンド テスト：Firestore Security Rules

バックエンドのテストは、Firestoreエミュレータを使用して隔離された環境で実行されます。技術的網羅性と視覚的証拠の両面から検証を行います。

### A. 網羅的ユニットテスト (Jest)
`@firebase/rules-unit-testing` を使用し、非表示フィールドの変更不可制約や、IDOR、権限昇格などの攻撃シナリオを50以上のケースで検証します。

-   **対象ファイル**: `tests/security_stub/verify_rules_authenticated.test.js`
-   **検証内容**: 
    -   自分のデータ以外の読み書き拒否
    -   特定のロール（Admin等）のみが可能な操作
    -   データのスキーマ/型バリデーション

### B. ビジュアル・セキュリティ・オーディット (Playwright)
実際のブラウザ環境で「セキュリティ・スタブ」を実行し、主要な5つの攻撃ベクトルに対する成否を視覚的なレポート（スクリーンショット）として生成します。

-   **対象ファイル**: `tests/security_stub/index.html` (Test Suites)
-   **ツール**: Playwright (Headless Browser)
-   **特徴**: 
    -   **自動ローテーション**: `tests/screenshots/` に最新10件のレポートのみを保持。
    -   **即時レポート**: 開発者以外（PMやQA）でも、PASS/FAILを視覚的に一目で確認可能。

### 🚀 実行コマンド
```bash
# セキュリティルールの網羅的検証とビジュアルレポート作成
npm run test:security:report
```

## 📱 2. フロントエンド テスト：Mobile App E2E (Maestro)

実機またはシミュレータ上で、実際の業務フローを自動操縦して検証します。

### A. フローベース検証
「ボタンを押して遷移したか」だけでなく、「Firestoreのデータが意図通りに変わったか」をセットで検証します。

-   **ツール**: Maestro
-   **構成**: `tests/jobs/*.yaml` にフロー定義。
-   **特徴**: 
    -   **I/O追跡**: 内部的なFirestoreの読み書き完了を待機し、非同期処理の不整合を排除。
    -   **エビデンス自動取得**: 成功・失敗時の動画/静止画を自動保存。

### B. 主要ディレクトリ
| ディレクトリ | 内容 |
| :--- | :--- |
| `tests/run_e2e.sh` | **メインエントリポイント**。環境構築からアプリ個別実行を制御。 |
| `tests/jobs/` | Maestro用のテストシナリオ (.yaml) |
| `tests/screenshots/` | テスト実行時のエビデンス（デバイス画面、セキュリティレポート） |

### 🚀 実行コマンド
```bash
# Adminアプリの全フロー検証
./tests/run_e2e.sh admin_app

# 個人ユーザーアプリの検証
./tests/run_e2e.sh individual_user_app
```

## 🛠 新しいテストを追加する手順

### セキュリティルールを強化した時
1.  `verify_rules_authenticated.test.js` に新しい `describe/it` ケースを追加。
2.  `index.html` の `runAllSecurityTests()` 関数内に、該当する新しい攻撃ペイロードと期待値を追記。

### 新機能を追加した時 (E2E)
1.  `tests/jobs/` に、ユーザーの行動に基づいた新しい `.yaml` シナリオを作成。
2.  単なる「表示されていること」の確認ではなく、**「データの状態が変わること」**を `assertVisible` 等で確認してください。
    -   **Bad**: `assertVisible: "完了"` (表示されただけで、中身が古い可能性がある)
    -   **Good**: `tapOn: "保存"` -> `assertVisible: "保存しました"` -> ホームに戻って「最新の値」が表示されているか確認。

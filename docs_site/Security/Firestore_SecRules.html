ABAC & RBAC Hybrid Rules

Custom Claims による高速な権限判定と、ドキュメント属性に基づく厳密なアクセス制御を兼ね備えた堅牢なデータ保護設計。

開発環境における一時的な全許可設定 (2026-02-12)

現在、開発効率を最優先するため、`allow read, write: if true;` を一時的に適用しています。本ドキュメントに記載されているルールは、リリース時に復元すべき 「あるべき姿 (Target State)」 です。

セキュリティ戦略

Hybrid Auth

Custom Claims (`request.auth.token.role`) を優先し、DB への `get()` 呼び出し（コストと遅延）を最小化します。 

function hasRole(role) {
  return request.auth.token.role == role
    || get(/.../users/$(uid)).data.role == role;
}

PII 分離 (Isolation)

個人情報 (PII) は `private_info` コレクションに隔離し、マッチングした企業のみに「許可リスト」形式で Read 権限を付与します。 

match /private_info/{userId} {
  allow read: if isOwner(userId) 
    || isMatched(resource.data.allowed_companies);
}

Key Improvements (2026-02-10)

データ整合性バリデーション

`public_profile` において `name` フィールドの必須化・非空文字チェックを実装。`FeeMgmtAndJobStatDB` において `amount` の正数検証を追加しました。

RBAC ベース基盤の確立

`users` コレクションへの不正なロール（Admin等）の自己付与をルールレベルで禁止。アプリケーションの信頼性を根幹から支えます。

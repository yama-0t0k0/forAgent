Firestore Security Optimization

"allow read, write: if true" から卒業し、本腰を入れたセキュリティへ

Project ID: flutter-frontend-21d0a Modern Security Standard
⚠️

Current Vulnerability Status

現在の設定は「誰でも読み書き自由」の、極めて危険な状態です。早急な対応が必要な 4 つのフェーズを提案します。

🚨 Current Rules (Vulnerable)

現在設定されているルールです。認証なしで誰でも全データの読み書き・削除が可能です。

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if true; // ⚠️ 危険
    }
    match /{coll}/{doc} {
      allow create, read, update, delete: if true;
    }
  }
}
01
Authentication Enforcement

第一歩として、ログインしていないユーザーからのアクセスを遮断します。OAuth（Firebase Auth）を必須化します。

allow read, write: if request.auth != null;
02
Resource Ownership Check

ユーザー本人のデータのみを操作できるように制限します。エンジニア登録情報（individual）などで必須です。

// 自分の UID とパスが一致する場合のみ許可
match /individual/{userId} {
  allow read, write: if request.auth.uid == userId;
}
03
Fine-grained Permissions

求人情報（job_description）や企業情報（company）など、情報の種類に応じた権限分離を行います。

// 求人は誰でも閲覧できるが、作成は企業/管理者のみ
match /job_description/{companyId}/JD_Number/{jdId} {
  allow read: if request.auth != null;
  allow write: if isCorporate(companyId) || isAdmin();
}
04
Schema Validation

許可されたユーザーであっても、想定外のデータ構造を送れないようにします。型や必須項目のチェックを導入します。

allow create: if request.resource.data.name is string
  && request.resource.data.age >= 18;

App Specific Access Policy

現在のアプリ構成に基づいた、推奨される各コレクションの制御方針です。

individual

エンジニア個人情報。本人のみが更新可能。管理者が読み取り可能。

Strict Ownership
company / corporate

企業プロフィール。企業本人が更新。全ユーザーが閲覧可能。

Public Read / Owner Write
job_description

求人票。企業・管理者が作成。エンジニアが閲覧。

Split Role Access
FeeMgmtAndJobStatDB

手数料・統計。管理者のみがアクセス可能。一般ユーザーは拒否。

Admin Only

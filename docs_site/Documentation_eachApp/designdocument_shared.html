共有モジュール
shared library

プロジェクト全体で再利用される、フロントエンド(Expo)およびバックエンド(Dart)の共通コードベース設計。

ディレクトリ構造

shared/ ├── common_frontend/ // Expo UI (shared/@shared/common-frontend) │ ├── src/core/ // State, Theme, Firebase, Models, Services │ └── src/features/ // Registration, Heatmap, Selection ├── common_backend/ // Dart Backend Infra ├── common_logic/ // プラットフォーム非依存 Dart Logic ├── domain_logic/ // ヒートマップ計算エンジン (Dart/JS) └── domain_models/ // DTO & JSON エンティティ定義

Common Frontend (Core)

State Management

`DataContext` による一元化された深い階層データのリアクティブ更新。React Context 配信。

updateValue(['基本情報', '氏名'], '...')

Data Models

`User`, `JobDescription`, `Company`, `SelectionProgress` クラス。分割管理されたデータの統合ロジックを内包。

user.fullNameKanji // getter

Services

`FirestoreDataService` による型安全なデータ取得。生 JSON ではなく、モデルインスタンスのリストを返却。

fetchAllIndividuals() -> Promise<User[]>

ロジック & 設計方針

🆕 Pure Registration 隔離アーキテクチャ

`yama` ブランチの大規模リファクタリングの影響を受けない「登録のみ」に特化したモードをサポート。

PureRecursiveField.js

モデルやテンプレート加工ロジックを排除。JSON構造そのままを再帰レンダリング。

PureRegistrationScreen.js

App Mode = "registration" 時のみ使用される、孤立したエントリポイント。

共通UI: ヒートマップ

`HeatmapGrid.js` は、`HeatmapCalculator` / `HeatmapGeometry` / `HeatmapMapper` 配下の共有ロジックを利用し、Individual/Job/Admin の各アプリで矛盾のない可視化を提供します。

開発ガイドライン

Frontend (Expo)

  • パスエイリアスの徹底: 相対パスは禁止。@shared @core 等を必ず使用。
  • モデル利用: FirestoreDataService から受け取ったインスタンスを、UI層で再変換しない。
  • 定数化: 文字列リテラルを避け、src/core/constants 配下で一元管理。

Backend (Dart)

  • domain_models: 全エンティティの唯一の情報源。build_runner でシリアライズコード生成。
  • common_logic: 非同期処理やバリデーションなど、プラットフォーム非依存の純粋ロジック。
dart run build_runner build
